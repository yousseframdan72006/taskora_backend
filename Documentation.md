# الدليل الشامل لمشروع Taskora (SaaS Backend) 🚀

هذا المستند يعتبر المرجع التقني الكامل والشامل (Documentation) للواجهة الخلفية (Backend) لمشروع **Taskora**. يشرح هذا الدليل المنطق التقني، معمارية النظام، دورة حياة البيانات، وتفاصيل جميع مسارات (APIs) النظام بمدخلاتها ومخرجاتها.

---

## 🧠 1. المنطق البرمجي ومعمارية النظام (Project Logic & Architecture)

مشروع **Taskora** هو تطبيق لإدارة المشاريع والمهام يعمل بنظام البرمجيات كخدمة **(SaaS)** ومبني على معمارية **المستأجرين المتعددين (Multi-tenant Architecture)** باستخدام ما يُعرف بـ `Workspaces` (مفساحات العمل).

### 🏢 نظام مساحات العمل وعزل البيانات (Workspace & Data Isolation)
- **الـ Global Scope:** لضمان الخصوصية التامة بين الشركات المختلفة، يعمل النظام بنطاق عام (Global Scope) يُسمى `WorkspaceScope`.
- **عزل البيانات:** أي كيان في النظام (مشروع `Project`، مهمة `Task`، تعليق `Comment`، أو نشاط `Activity`، دعوة `Invite`، إلخ) يمتلك حقل `workspace_id`.
- بمجرد تسجيل دخول المستخدم، يتحقق النظام من مساحة العمل التابع لها (`$user->workspace_id`). ويتكفل إطار العمل تلقائياً بفلترة جميع الاستعلامات بحيث لا تُرجع أو تُعدّل إلا البيانات التي تخص مساحة العمل تلك. من المستحيل تقنياً أن يتسرب أي بيانات من مساحة عمل إلى أخرى.
- يمكن للمستخدم أن يكون مستقلاً (Independent) بدون مساحة عمل، وعندها تنحصر صلاحياته في إنشاء مساحة عمل جديدة أو قبول دعوة للانضمام إلى مساحة عمل قائمة.

### 👥 نظام الدعوات والانضمام (Invitations Flow)
1. **الإرسال:** يقوم مدير مساحة العمل (Admin) بدعوة موظفين جدد عبر البريد الإلكتروني مع تحديد صلاحيتهم (Admin أو Member).
2. **الاستلام:** يُسجل النظام الدعوة بوضع `pending`. إذا كان المدعو يمتلك حساباً في النظام مسبقاً، يصله **Push Notification** فوري.
3. **القبول/الرفض:** يستعرض المدعو الدعوات الموجهة إليه ويقرر قبولها أو رفضها. عند القبول، يتم ربطه تلقائياً بمساحة العمل ويُحدث حقل `workspace_id` و الـ `role` الخاص به في جدول المستخدمين.

### 🗂️ هيكلة المشاريع، المهام والأدوار (Projects, Tasks & Roles)
- **المشاريع (Projects):** الحاوية الكبرى للعمل. يمكن إضافة أعضاء للمشروع كاملاً لتحديد فريق العمل المسموح له بالوصول وتلقي الإشعارات العامة.
- **المهام (Tasks):** تُدرج داخل المشاريع، وتمتلك خصائص مفصلة (العنوان، الوصف، الأولوية، تاريخ الاستحقاق، والحالة).
- **إدارة الأدوار المرنة (Flexible Roles):** لا تُسند المهمة لشخص واحد فقط، بل تُسند عبر جدول وسيط `task_user` يسمح بتعيين عدة مستخدمين على نفس المهمة بأدوار مختلفة (مثال: مستخدم بصفة `developer` ومستخدم آخر بصفة `tester`). هذا يسمح بمرونة عالية في تقسيم العمل.

### 🛤️ سير العمل وتتبع التغييرات (Workflow & Audit Trail)
- **آلة الحالة (State Machine):** تخضع حالات المهمة لقيود صارمة وتدفق منطقي إجباري (`pending/todo` ➔ `in_progress` ➔ `review` ➔ `done`). لا يمكن تخطي مرحلة بشكل عشوائي، مما يضمن التزام الفريق بسير العمل المعتمد.
- **سجل التتبع الدقيق (Audit Trail):** كل إجراء يتم على المهمة (تغيير حالتها، تعديل تفاصيلها، أو إسناد أشخاص جدد إليها) يتم التقاطه تلقائياً خلف الكواليس عبر نظام أحداث (Events) ويُسجّل في جدول `task_activities`. هذا يوفر سجلاً تاريخياً (History) كاملاً لكل ما حدث في المهمة ومن قام به ومتى.

## 🔔 2. نظام الإشعارات الهجين (Hybrid Push Notification System)

يستخدم Taskora نظام إشعارات متقدم لضمان بقاء أعضاء الفريق على اطلاع فوري بالتحديثات، ويعمل بنموذج "هجين" يجمع بين:

1. **الإشعارات اللحظية عبر Firebase (FCM HTTP v1):** 
   - تُرسل مباشرة إلى جهاز المستخدم (Android, iOS, Web) لتنبيهه حتى وإن كان التطبيق مغلقاً.
   - يعتمد على تسجيل توكنات الأجهزة (Device Tokens) في جدول `user_devices`، ويدعم امتلاك المستخدم لأكثر من جهاز (Multi-device support).
   - يتم بناء وتوقيع JWT واستخراج OAuth2 Token مؤقت للاتصال بـ Firebase بشكل آمن ومستقل تماماً عن الباقات الخارجية المعقدة.

2. **سجل الإشعارات داخل التطبيق (In-App Database Notifications):**
   - يُخزن كل إشعار في جدول `notifications` في قاعدة البيانات.
   - يوفر للمستخدم مركز إشعارات داخل التطبيق (Notification Center) لعرض التاريخ الكامل للإشعارات مع تتبع حالتها (مقروءة أو غير مقروءة).

### ⚙️ كيف تعمل الإشعارات خلف الكواليس؟
- **هندسة مبنية على الأحداث (Event-Driven):** عند حدوث فعل في الـ Controllers، يتم إطلاق (Dispatch) حدث (Event).
- **المعالجة غير المتزامنة (Async via Queue):** يقوم مستمع الحدث (Listener) بتجميع الفئة المستهدفة وتمريرها إلى وظيفة (Job) اسمها `SendPushNotification` تُنفذ في الـ Queue (الطابور). هذا يضمن أن الإرسال عبر الإنترنت لـ Firebase لا يبطئ استجابة الـ API للمستخدم.
- **قاعدة عدم إزعاج الفاعل (No Self-Notification):** النظام مصمم بذكاء بحيث يُستثنى المستخدم الذي قام بالحدث من تلقي إشعار عن فعله الخاص (مثلاً: إذا قمت أنت بإضافة تعليق، لن يصلك إشعار بأنك أضفت تعليقاً).

### 🎯 الأحداث التي تولد إشعارات (Triggered Events)

> **القاعدة:** كل فعل يحدث من أي طرف (Admin أو Member) يصل الطرف الآخر إشعار فوري.

| الإجراء (Action) | من يُنفّذه | الحدث المُطلق (Event) | المستهدفون (Recipients) | نوع الـ data |
| :--- | :---: | :--- | :--- | :--- |
| **إرسال دعوة انضمام** | Admin | `InviteSent` | المدعو (إن كان لديه حساب) | `invite_received` |
| **قبول الدعوة** | Member | `InviteResponded` | الـ Admin | `invite_response` |
| **رفض الدعوة** | Member | `InviteResponded` | الـ Admin | `invite_response` |
| **إنشاء مشروع** | Admin | `ProjectCreated` | جميع أعضاء المشروع (باستثناء المنشئ) | `project_created` |
| **إضافة أعضاء لمشروع** | Admin | (مباشر في Controller) | الأعضاء المضافين | `project_assigned` |
| **تعديل مشروع** | Admin | `ProjectUpdated` | جميع أعضاء المشروع (باستثناء المعدّل) | `project_updated` |
| **حذف مشروع** | Admin | `ProjectDeleted` | جميع أعضاء المشروع السابقين | `project_deleted` |
| **إزالة عضو من مشروع** | Admin | `MemberRemovedFromProject` | العضو المُزال | `removed_from_project` |
| **إزالة عضو من الـ Workspace** | Admin | `MemberRemovedFromWorkspace` | العضو المُزال | `removed_from_workspace` |
| **إسناد مهمة لمستخدم** | Admin/Member | `TaskAssigned` | المستخدم الذي تم تعيينه | `task_assigned` |
| **تعديل تفاصيل المهمة** (عنوان/أولوية/إلخ) | Admin/Member | `TaskUpdated` | جميع المشاركين في المهمة (باستثناء المعدّل) | `task_updated` |
| **تغيير حالة المهمة → `review`** | Member | `TaskStatusChanged` | جميع المستخدمين بدور `tester` | `task_status_changed` |
| **تغيير حالة المهمة → `done`** | Member | `TaskStatusChanged` | جميع المشاركين في المهمة | `task_status_changed` |
| **تغيير حالة أخرى للمهمة** | Admin/Member | `TaskStatusChanged` | جميع المشاركين (باستثناء المغيّر) | `task_status_changed` |
| **حذف مهمة** | Admin | `TaskDeleted` | جميع المشاركين في المهمة السابقين | `task_deleted` |
| **إضافة تعليق** | Admin/Member | `CommentAdded` | جميع المشاركين في المهمة (باستثناء كاتب التعليق) | `comment_added` |

## 📡 3. تفاصيل الـ APIs (مسارات النظام، المدخلات، والدور)

فيما يلي التوثيق الشامل لجميع الـ APIs المتاحة في النظام، مرتبة حسب مجموعاتها الوظيفية.

### 🔐 3.1. المصادقة وإنشاء الحساب (Authentication)
*هذه المسارات عامة ولا تتطلب توكن (Public)*

*   `POST /api/auth/register`
    *   **الدور:** تسجيل حساب مستخدم جديد في النظام.
    *   **المدخلات (Body JSON):** `name`, `email`, `password`, `password_confirmation`.
*   `POST /api/auth/login`
    *   **الدور:** التحقق من الهوية وتسجيل الدخول وإصدار توكن الوصول.
    *   **المدخلات (Body JSON):** `email`, `password`.
    *   **المخرجات:** يرجع بيانات المستخدم مع الـ Bearer Token (Sanctum).

### 👤 3.2. إدارة الحساب الشخصي (Profile & Session)
*تتطلب `Authorization: Bearer {token}`*

*   `GET /api/auth/me`
    *   **الدور:** استرجاع البيانات الكاملة للمستخدم الموثق الحالي مع مساحة العمل المرتبط بها.
*   `POST /api/auth/logout`
    *   **الدور:** إتلاف التوكن الحالي وتسجيل خروج المستخدم من الجهاز.

### 📱 3.3. إدارة أجهزة المستخدم للإشعارات (Device Tokens)
*هذه المسارات مسؤولة عن ربط أجهزة المستخدم بنظام Firebase FCM*

*   `POST /api/devices/token`
    *   **الدور:** تُستدعى من الموبايل (Flutter) عند **تسجيل الدخول** أو عند **تشغيل التطبيق** لتسجيل الـ FCM Token الخاص بالجهاز.
    *   **المدخلات (Body JSON):** 
        *   `device_token` (مطلوب): التوكن المستخرج من Firebase SDK.
        *   `platform` (اختياري، الافتراضي android): نظام تشغيل الجهاز (`android`, `ios`, `web`).
*   `DELETE /api/devices/token`
    *   **الدور:** تُستدعى من الموبايل عند **تسجيل الخروج** لإزالة التوكن من قاعدة البيانات وإيقاف الإشعارات لهذا الجهاز.
    *   **المدخلات (Body JSON):** `device_token` (مطلوب).

### 🏢 3.4. مساحات العمل والدعوات (مستوى المستخدم المبدئي)
*مسارات خاصة بالمستخدم قبل انضمامه لبيئة عمل فعلية أو لإنشاء واحدة.*

*   `POST /api/workspaces`
    *   **الدور:** إنشاء مساحة عمل/شركة جديدة ليصبح المستخدم الحالي هو المالك/المدير (Admin) لها.
    *   **المدخلات (Body JSON):** `name` (اسم مساحة العمل).
*   `GET /api/invites/pending`
    *   **الدور:** عرض جميع دعوات الانضمام المعلقة المُرسلة لعنوان البريد الإلكتروني للمستخدم الحالي.
*   `POST /api/invites/{invite_id}/accept`
    *   **الدور:** قبول دعوة الانضمام ليصبح المستخدم عضواً في مساحة العمل الخاصة بالدعوة.
*   `POST /api/invites/{invite_id}/decline`
    *   **الدور:** رفض الدعوة.

---
**⚠️ ملاحظة هامة:** جميع الـ APIs التالية تخضع لـ Middleware `EnsureWorkspace` الذي يتأكد أن المستخدم الحالي منضم بنجاح إلى مساحة عمل.
---

### ⚙️ 3.5. إدارة مساحة العمل والدعوات (Workspace Admin Level)

*   `GET /api/workspaces/details`
    *   **الدور:** عرض تفاصيل مساحة العمل الحالية الخاصة بالمستخدم.
*   `PUT /api/workspaces/update`
    *   **الدور:** تحديث إعدادات أو بيانات مساحة العمل (مقتصرة على الإدارة غالباً).
    *   **المدخلات (Body JSON):** `name`.
*   `GET /api/workspaces/dashboard`
    *   **الدور:** توفير بيانات إحصائية وملخص سريع يظهر في واجهة لوحة تحكم الإدارة لمساحة العمل.
*   `POST /api/invites`
    *   **الدور:** توجيه دعوة انضمام لموظف جديد عبر بريده الإلكتروني.
    *   **المدخلات (Body JSON):**
        *   `email`: البريد الإلكتروني للمدعو.
        *   `role`: الصلاحية المقترحة للمدعو داخل الشركة (`admin` أو `member`).
    *   **تنبيه:** يطلق الحدث `InviteSent` ويرسل إشعاراً إذا كان المدعو يملك حساباً بالفعل.

### 🗂️ 3.6. إدارة المشاريع (Projects)

*   `GET /api/projects`
    *   **الدور:** استعراض قائمة بجميع المشاريع المُسجلة تحت مساحة العمل الحالية، مع عرض الأعضاء المنضمين للمشاريع.
*   `POST /api/projects`
    *   **الدور:** فتح/إنشاء مشروع جديد داخل مساحة العمل. يقوم النظام تلقائياً بتعيين مُنشئ المشروع كـ (admin) عليه.
    *   **المدخلات (Body JSON):** `name`, `description`, `status`, `due_date`.
    *   **تنبيه:** يطلق الحدث `ProjectCreated` ويرسل إشعاراً لكل المعينين عليه (غير المنشئ).
*   `GET /api/projects/{project_id}`
    *   **الدور:** عرض البيانات التفصيلية لمشروع معين تشمل المهام (Tasks) وقائمة المستخدمين (Users) العاملين عليه.
*   `PUT /api/projects/{project_id}`
    *   **الدور:** تعديل خصائص المشروع الحالي.
    *   **المدخلات (Body JSON):** (نفس حقول الإنشاء وتكون اختيارية).
*   `DELETE /api/projects/{project_id}`
    *   **الدور:** حذف المشروع بالكامل.
*   `POST /api/projects/{project_id}/assign`
    *   **الدور:** إضافة مستخدمين كأعضاء في المشروع، مما يمنحهم رؤية أعم للمشروع ومهامه.
    *   **المدخلات (Body JSON):**
        *   `user_ids`: مصفوفة (Array) بمعرفات المستخدمين (UUIDs).
        *   `role`: دورهم في المشروع.

### ✅ 3.7. إدارة المهام (Tasks)

*   `GET /api/tasks`
    *   **الدور:** جلب المهام التابعة لمساحة العمل. يمكن تمرير بارامتر الفلترة.
    *   **المدخلات (Query Params):** `project_id` (اختياري) لجلب مهام مشروع محدد.
*   `POST /api/tasks`
    *   **الدور:** استحداث مهمة جديدة وربطها بمشروع.
    *   **المدخلات (Body JSON):** `title`, `description`, `status`, `priority`, `due_date`, `project_id` (مطلوبة).
*   `GET /api/tasks/{task_id}`
    *   **الدور:** جلب المهمة بكامل متعلقاتها (المشروع المرتبطة به، المستخدمين بأدوارهم، سجل التعليقات بالكامل، وصاحب كل تعليق).
*   `PUT /api/tasks/{task_id}`
    *   **الدور:** تحديث المهمة. (وهنا يُطبق المنطق البرمجي الخاص بـ State Transitions).
    *   **المدخلات (Body JSON):** يمكن تحديث `title`, `description`, `status`, `priority`, `due_date`.
    *   **تنبيه:** إذا شمل التعديل حقل `status`، سيطلق النظام حدث `TaskStatusChanged` ويوزع الإشعارات حسب القواعد (للمختبرين أو للمشاركين إلخ)، ويُسجل الحدث في `task_activities` للـ Audit Trail.
*   `POST /api/tasks/{task_id}/assign`
    *   **الدور:** إسناد مهمة لعضو وتحديد دوره الوظيفي بداخلها.
    *   **المدخلات (Body JSON):** 
        *   `user_id`: معرف المستخدم (UUID).
        *   `role`: نوع المهمة المُسندة (`admin`, `developer`, `tester`, `designer`).
    *   **تنبيه:** يطلق الحدث `TaskAssigned` لتنبيه المستخدم المعيّن فوراً (FCM Push) ويسجل العملية في `task_activities`.
*   `DELETE /api/tasks/{task_id}`
    *   **الدور:** حذف المهمة.

### 💬 3.8. التعليقات (Comments)

*   `POST /api/tasks/{task_id}/comments`
    *   **الدور:** إرفاق تعليق على المهمة لمناقشة تفاصيل العمل.
    *   **المدخلات (Body JSON):** `content` (نص التعليق).
    *   **تنبيه:** يطلق الحدث `CommentAdded` ويرسل إشعاراً لبقية الأعضاء في المهمة.
*   `DELETE /api/comments/{comment_id}`
    *   **الدور:** مسح تعليق محدد (يخضع لـ Policies وصلاحيات صارمة).

### 🔔 3.9. مركز الإشعارات (In-App Notifications)

*   `GET /api/notifications`
    *   **الدور:** جلب قائمة بالإشعارات الخاصة بالمستخدم (Pagination). تعرض تاريخ الإشعارات وأيها لم يُقرأ.
*   `PUT /api/notifications/read-all`
    *   **الدور:** تحديد كافة الإشعارات غير المقروءة للمستخدم وتحويل حالتها إلى "مقروءة" دفعة واحدة.
*   `PUT /api/notifications/{notification_id}/read`
    *   **الدور:** تعليم إشعار واحد محدد على أنه مقروء.

### 📊 3.10. لوحات التحليل والإحصائيات (Analytics & Dashboards)

*   `GET /api/dashboard/task-overview`
    *   **الدور:** يزود واجهة المستخدم الرئيسية بإحصائيات المهام المجمعة (المهام قيد التنفيذ، المتأخرة، المنجزة)، ويحدد من هو الموظف الأكثر ضغطاً بالمهام (Most loaded user).
    *   *ملاحظة:* النظام يستخدم Caching داخلي لتسريع هذا المسار وتقليل العبء على قاعدة البيانات.
*   `GET /api/workspaces/employees/task-stats`
    *   **الدور:** يوفر للإدارة تقريراً مفصلاً عن أداء الموظفين، نسب إنجازهم (Completion Rate)، وعدد مهامهم المتأخرة لتسهيل عملية الإدارة والتقييم.
*   `GET /api/users/{user_id}/task-stats`
    *   **الدور:** يرجع إحصائيات الأداء لموظف واحد محدد (يُسمح للموظف باستعراض إحصائياته الخاصة، ويُسمح للمدير باستعراض إحصائيات أي موظف).

## 🗄️ 4. هيكل قاعدة البيانات المبسط (Database Schema)

نظرة عامة على جداول قاعدة البيانات ودورها:

| اسم الجدول | الوصف |
| :--- | :--- |
| `users` | يخزن بيانات الحسابات، ويربط المستخدمين بمساحات العمل (`workspace_id`) مع تحديد الصلاحية الكلية (`role`). |
| `workspaces` | سجلات الشركات أو مساحات العمل المنفصلة. |
| `user_devices` | يخزن توكنات Firebase (FCM) لكل مستخدم لدعم الإشعارات اللحظية لعدة أجهزة. |
| `projects` | سجلات المشاريع المرتبطة بمساحات العمل. |
| `project_user` | جدول وسيط (Pivot) يربط المستخدمين (أعضاء الفريق) بالمشاريع المنضمين لها. |
| `tasks` | سجلات المهام. ترتبط بمشروع (`project_id`) ومساحة عمل (`workspace_id`). |
| `task_user` | جدول وسيط يربط المستخدمين بمهامهم مع تحديد الدور الوظيفي المخصص لهم في المهمة. |
| `task_activities` | سجل المراقبة (Audit Trail) يخزن كل تغيير يحدث للمهمة ومتى حدث ومن قام به. |
| `comments` | تعليقات فرق العمل داخل المهام. |
| `notifications` | سجل الإشعارات الداخلية للمستخدم، يعتمد على علاقة `morphTo` (Polymorphic). |
| `invites` | سجل الدعوات المُرسلة للموظفين الجدد بحالاتها (معلقة، مقبولة، مرفوضة، منتهية). |
| `jobs` | جدول خلفي لتشغيل المهام غير المتزامنة (Queue) مثل إرسال دفعات إشعارات Firebase في الخلفية. |
